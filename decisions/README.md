# Decision Log

An append-only record of consequential decisions for Tollwing, following the
[ADR pattern](https://cognitect.com/blog/2011/11/15/documenting-architecture-decisions) (Michael Nygard, 2011) as refined by [MADR](https://adr.github.io/madr/). Adopted in [DEC-001](DEC-001-adopt-constitution-and-governance.md).

A decision here is not the same as a principle. **Principles** ([`../CONSTITUTION.md`](../CONSTITUTION.md)) are the binding rules. **Decisions** are the append-only history of consequential choices made *under* those rules — with the alternatives that were weighed and why they lost. Decisions are never deleted; they are superseded or deprecated, with the status edited in place and the superseding ADR linked.

The audience explicitly includes **AI agents**: an ADR is how a future agent inherits the reasoning behind a choice instead of re-litigating it.

## When to record a decision

Write an ADR when a choice is any of:

1. **Architecturally consequential** — affects a subsystem boundary, the data plane, storage, or the agent/control-plane split.
2. **Cross-cutting** — touches multiple packages or every contributor.
3. **A reversal** — supersedes or deprecates an earlier decision.
4. **A constitutional exception** — knowingly accepts a principle violation (the ADR *is* the exception; code cites it as `// Per DEC-NNN, …`).
5. **A constitutional amendment** — changes a principle's text (record the amendment as an ADR).
6. **A sequencing/prioritization call with real alternatives** — where the alternatives deserve to be written down.
7. **Hard to reverse** — a one-way door (a storage format, a wire protocol, a kernel-hook strategy).
8. **A constraint on future work** — it forecloses options a later contributor might otherwise try.

When in doubt, record it. A short ADR is cheap; a re-litigated decision is not.

## What NOT to record

- Trivial implementation choices obvious from the code.
- Routine task completion ("added a test", "fixed a typo").
- Tactical, local choices with no cross-cutting impact and no real alternative.
- Anything already fully explained by a code comment that nothing else depends on.

## How to add one

1. Copy [`TEMPLATE.md`](TEMPLATE.md) to `DEC-NNN-{kebab-slug}.md` (next sequential `NNN`, zero-padded to 3 digits; then 4 if we ever exceed 999).
2. Fill it in — the **Alternatives considered** and **Constitutional principles touched** sections are mandatory and load-bearing.
3. Run `go run ./tools/governance index` to regenerate the index table above.
4. If the decision implements an existing ADR, say so in the commit (`feat(servicegraph): … (DEC-006)`).

| Tool | What it does | When to run |
|---|---|---|

| `go run ./tools/governance scan` | Scans `pkg/`+`cmd/` for mechanically-detectable principle violations (P6/P7/P9) | Before a PR; pre-commit hook |
| `go run ./tools/governance scan -gate` | Fails on blocking-class violations only | In CI (the `governance` job) |
| `go run ./tools/governance audit -principle Pn` | Generates an audit prompt to paste into an AI agent | When running an audit ([playbook](../docs/governance/audit-playbook.md)) |
| `tools/governance/install-hooks.sh` | Points git at `.githooks/` (warn-only pre-commit scan) | Once, after cloning |

## How agents should use this log

Before significant work, scan the index for decisions that bind the area you're touching, and read those ADRs. After significant work, if the change warranted a decision (see "When to record"), write the ADR from the template and regenerate the index *in the same PR*. See [`../CLAUDE.md`](../CLAUDE.md).

## How humans should use this log

The index is the table of contents. Read DEC-001 for the governance system itself; read the ADR for any subsystem before you change its fundamentals. The "Alternatives considered" section is usually the most valuable part.
